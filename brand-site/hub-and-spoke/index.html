The Model

One brain. Isolated servers.

Every client gets their own isolated server. One central hub holds everything that makes us us — templates, processes, voice profiles, env configs, and shared learnings. Each client server pulls from that hub automatically.

Think of it like a franchise model: the playbook lives at HQ, but each location runs independently.

Security & Trust

Client A's data never touches Client B's environment. No accidental leaks, no shared credentials, no crossed wires. If a client ever asks "who has access to our data?" — the answer is clean.

Simplicity

Each server is self-contained. No complex folder permissions, no "make sure you're in the right directory." You open the client's server, everything in front of you is theirs.

Scalability

Adding a new client = spinning up a new server. Removing a client = shutting one down. No untangling dependencies.

Parallel Work

Joshua can be working on Jason's server while Ricki is building automations on CrossCourt's. No merge conflicts, no stepping on each other's toes.

Audit Trail

Each server has its own git history. Every change is tracked per-client. Clean records if anyone ever needs to look back.

The Spokes

Client servers

Each client server is its own workspace with a standard structure and a live link to the hub.

client-[name]/ ├── CLAUDE.md # client-specific instructions ├── .env # client API keys & creds ├── hub/ # ← git submodule (auto-synced) │ └── (entire hub repo) ├── memory/ │ ├── MEMORY.md │ ├── client_voice_profile.md # their brand voice │ ├── client_context.md # org, goals, stakeholders │ └── project_notes.md # ongoing project context ├── deliverables/ │ └── (whatever we're building) └── assets/ └── (brand assets, docs, etc.)

Each client's CLAUDE.md includes: "Read from ./hub/ for agency-wide processes, templates, and voice profiles. Apply the agency playbook, then layer in this client's specific context."

Claude Code on any client server automatically knows our full agency playbook AND the client's specific world.

Onboarding

New client setup

One command kicks off the entire setup when we sign a new client.

./spin-up-client.sh [client-name]
1

Spin up the server

Provisions a new Hetzner cloud environment for the client.

2

Install the toolchain

Claude Code (AI operating layer), OpenCrawl (web scraping/auditing), and any other standard tools.

3

Clone the hub as a submodule

Pulls in all agency templates, playbooks, voice profiles, and shared .env variables. Immediate operational context.

4

Scaffold the workspace

Creates the standard folder structure (memory/, deliverables/, assets/), starter CLAUDE.md, and placeholder memory files.

5

Copy shared .env values

API keys for our tools (Anthropic, Cynthia platform, etc.) get pulled from the hub automatically.

After the script runs, either of us can open that server from any computer and start working immediately. The client is fully operational within minutes, not hours.

Sync

How it stays in sync

Automatic

Hub → Spoke

Each time we open a client server or run sync-hub.sh, the latest hub gets pulled in. New templates, updated processes, fresh learnings — they flow to every client automatically.

Intentional

Spoke → Hub

When we learn something on a client project that applies universally, we deliberately push it to the hub. We curate what goes upstream — client-specific details never leak into the shared brain.

This is how the agency gets smarter with every client. The hub is a living, evolving playbook — not a static template library.

Access

Both of us, anywhere

Cloud servers mean the server IS the source of truth — not our laptops.

  • Joshua can work from his laptop in San Diego
  • Ricki can work from his setup wherever he is
  • Both can work on the same client simultaneously if needed
  • All changes tracked via git — full history of who did what
  • Picking up where the other left off is seamless
  • Claude Code's memory persists on the server, not on local machines

No "can you send me that file?" No "which version is current?"

Summary

TL;DR

  • One hub = our agency brain (templates, processes, learnings, shared config)
  • One server per client = isolated, secure, simple
  • Hub syncs to all spokes = every client gets our latest and greatest
  • Universal learnings flow back to hub = the agency gets smarter with every engagement
  • One script to onboard = new client goes from signed to operational in minutes
  • Either of us, any computer = Hetzner cloud servers mean location doesn't matter

The architecture grows with us. Client #3 benefits from everything we learned on clients #1 and #2. Client #10 benefits from everything before it. The playbook compounds.
